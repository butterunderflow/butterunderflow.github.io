Partial Evaluation Techniques

Posted on 2022-11-19 Edited on 2022-11-21

partial evaluation 的第四章 Partial Evaluation for a Flow Chart Language 的笔记.

这一章介绍了一个叫做 Flow Chart 的语言, 是一个以基本块组成的语言. 然后通过两个二村映射来介绍partial evaluation的技巧.

然后就是程序点和程序状态的概念: 程序点就是程序执行的位置, 程序状态指的是程序执行的状态, 具体的状态的值域取决于语义的定义. 这一点和静态分析里的定义是类似的.

程序的状态可能有很多, 但大部分的语言都可以通过 变量 - 值 的映射来表示. 其中对于每个 program point, 变量的值可以分为 static 和 dynamic 分为两类. static 表示变量的值可以在静态确定, 而 dynamic 的值只能在运行时确定. 将每个 program point 的变量区分为 static 和 dynamic 的过程叫做 division.

这就引出了一个概念叫做 poly, 指的是基于输入和已知的程序点的 static 变量的值, 可以最大确认多少程序点的 static 变量? 可以通过一个由 程序点-static value 序对组成的集合来表示 poly. poly 对应着 residual program (specialized program point) 的 program point. 一个直觉的描述是: 对于 subject program 在每个 program point, 把这个 program point 对应的 basic block 根据不同的 static state "展开"的结果就是 residual program; "展开"之后, static 的属性会"嵌入"到 residual program 中, 而不是通过 division 来表达.

division 的结果并不一定唯一, 只要满足以下 congruent 条件就行: > any variable that depends on a dynamic variable must itself be dynamic.

这个定义其实有一些模糊. 比如说, 如果一个循环中的变量, 第一次循环的值是确定的, 然后在循环中会通过赋值更新这个变量(这个 Flow Chart 的语言是允许赋值的), 那么这个变量算是什么呢?

这可能又要分很多种情况, 比如说这个循环的循环次数和对变量更新的值都是可以确定的 (假设是 k 次), 那么我们就可以把这个循环展开 k 次, 展开后的程序的循环变量显然都是 static 的.

但是如果每次循环对变量更新的值是可以 static 确定的, 但是循环的次数可能有无限多次, 这个变量我们应该标记为 static 吗?

对于这种情况, 给定任意的 k, 我们都能确定这个变量第 k 次循环的值, 但是, 把他们当作 static 的, subject program 的 poly 将会是无限的, 从而无法有效的计算出一个 residual program (程序都是有限的). 所以, 从计算residual program的角度来说, 有无限多种可能的 static variable 是没有意义的.

所以我们只能把他们标记为 dynamic. 这样就有了一种最简单的处理方式: 所有麻烦的东西都当成 dynamic 就好了! 但是从 partial evaluation 最基本的直觉出发, 我们标注为 dynamic 的东西越多, 我们能够 specialize 的东西就越少, partial evaluation 所能带来的优化也就更少. 所以, 我们的 division 应该尽可能的让 dynamic 少.

那有没有什么通用的办法, 让我们给出 division 的同时, 还能保证这个 division 的 dynamic 是 最少 的?

这个问题是 不可判定 问题, 不过书上说他们之后会给个办法给一个可接受的解(前面提到了, 合理的 division 并不唯一). 现在先假设每个 program point 都有着相同的 division, 通过一个叫 BTA (Bind Time Analysis) 的分析来确认这个 division.

总结一下 partial evaluation 的步骤: 1. 根据输入, 确定变量的 division; 也就是把变量区分为 static 和 dynamic 两类
2. 计算 poly, 也就是 specialized program 的 program points
3. 根据 poly, 生成 specialized program
4. (optional) transition compression
5. (optional) relabel the program

Language and Interpreters

Posted on 2022-11-08 Edited on 2022-11-19

partial evaluation 的第三章 Programming Languages and Interpreters 的笔记.

这一章主要是形式化的介绍了一下程序, 解释和编译. 程序只是一种表示, 在这里程序的语义指的是 Operational Semantic, 代表了程序应该如何规约(reduction).

解释是指的用另外一个程序来实现程序的语义, 解释器本身也是在某种抽象机上(解释)执行的. 对于某个目标程序 P, 解释器 int 在对应抽象机上的解释 P 的执行步数, 一般要比 P 在 P 对应的抽象机上的执行步数要成倍数的增长. 比如说对于一个变量的求值, 在代表程序语义的抽象机上一共只需要一步; 而在解释器中, 会对应到5步:
1. 解释函数 (如 eval) 的调用
2. 模式匹配判断当前表达式是一个变量
3. 读取环境
4. 读取变量名
5. 读取环境中对应变量名的值

可以通过case by case的分析各种语义的对应步数得到一个大致的倍数, 从而估算解释的开销(overhead).

刚刚提到了开销, 这个开销实际上是不可以忽略的, 首先在真实世界的编程语言中, 这个开销的倍数可能非常大(比如说可能是几十倍); 另一方面, 如果涉及到多层的解释, 那么这个开销的倍数也会相乘, 最终开销也会呈指数增长. 将源程序直接编译为可执行的程序(比如说机器指令)就是减少在开销上做乘法的一种思路(减少解释的开销).

然后介绍了一下编译器自举(bootstrapping)的概念, 自举大概指的是一个compiler h 可以通过被其compiled version t 编译得到其compiled version t 的. 书上通过举了个例子说明如何通过扩展 S 语言实现的 S 语言的编译器并完成自举的过程: 首先我们已有 S 语言编译器的源码 h 和编译后的代码 t 1. 然后我们在 h上扩展, 得到了一个 S'的编译器 h' 2. 然后用一开始的 t 把 h' 编译, 得到 t1' 3. 再用 t1' 编译一遍 h'. 这里第一次用了扩展后的编译器 h'的语义. t1' 也是h'的compiled program, 但因为 t1' 不是由h'的语义编译生成的(是由h的语义即 t 编译生成的). 所以这里还算不上自举 4. 最后用 t2'编译 h', 这里就有了自举了. 因为 t3' 是完全等于 t2'的, 因为 t1'的语义和 t2' 的语义相同.

Substructural Type System: Linear Type

Posted on 2022-10-17 Edited on 2022-10-23

Backgournd

Structural Property:

  1. Exchange: 环境中的binding可以任意交换顺序, 保持得到的类型命题(类似 x:T 的东西)不变
  2. Weakening: 给环境中增加binding不会影响类型命题
  3. Contraction: 把环境中相同类型的两个相邻binding x2和x3拿掉, 原地替换成另一个相同类型的binding x3, 然后把term中的x1和x2替换成x3, 不影响类型命题.

contraction严格定义:

substructural type system指的就是满足一条或多条structural property的类型系统. 可以通过满足的数量分为不同的类型系统, E(xchange), W(eakening), C(ontraction):

Linear Type

Linear Type是指的只满足Exchange的类型系统. 在linear type system中, 类型被一个quantifier标注, 标注一个类型是否是linear的, 由于不满足Weakening和Contraction, 具有linear的类型的值必定只能被使用一次: 1. 不满足Weakening: 不能随时通过"增强"环境创造类型命题 2. 不满足Contraction: 不能随时通过"裁剪"环境创造类型命题.

Linear Type Checker的实现

实现的直觉是这样的:

  1. 将先前的declarative的规则转换成algorithmic规则的时候有一个难点, 我们无法确定我们应该如何用 \(\circ\) 运算符来拆分一个环境, 使有linear type的variable只能出现在一个环境中. 从而导致不知道应该如何将问题拆分为子问题.

  2. 所以说拆分一个环境的最终目的是为了让linear type binding只出现在一个环境中, 来保证这个linear type的variable只会被使用一次.

那么可行的一种做法就是: 先不考虑lin type binding的分配问题, 而是先检查, 如果用到了一个lin变量就把environment中的这个lin变量消除, 这样如果碰到了使用两次lin变量的使用就无法type checked(也就是检查出了类型错误). 所以, 在一个environment中检查一个term, 不仅会会check一个term的类型, 还可能会对环境带来改变(通过返回一个新的环境).

Pointer Analysis Introduction

Posted on 2022-08-31 Edited on 2022-10-17

指针分析是什么?

指针分析尝试回答这样一个问题: 一个程序的一个指针在运行时的值是什么? 但是这个问题其实是"无法回答"的, 首先, 这个问题是不可判定问题; 其次, "指针的值是什么"这个问题, 它的回答应该满足什么形式无法确定. 回答这个问题大致有两种思路: 指向分析与别名分析. 指向分析用到了堆抽象的技术, 堆抽象提供了一种抽象的堆模型来描述指针值的指向信息, 通过使该模型满足特定的性质来保证这个问题成为可判定问题(可回答).

别名分析不尝试对指针的值做抽象, 而是尝试回答"哪些指针的值是相等的"这个问题来回答"指针的值是什么".

Heap Abstraction

如前所述, 堆抽象模型即要能表达"指向信息", 又要保证是有限的空间, 因为只有有限的才能保证一个指针分析算法停机. 比较简单的堆抽象模型是Alocation-Site Abstraction.

堆模型又可以分为store-based model和store-less model, 这两种模型都将堆通过图来表示, 不同的是: store-based中, 图的节点直接对应堆对象的创建(比如将所有的同一个位置的创建的所有对象映射到图的一个节点); store-less中, 图的节点不对应堆对象的创建, 而是对应一个AccessPath所有能访问到的堆对象.

Alocation-Site Abstraction(most used)

  • Model concrete objects by their allocation sites
  • One abstract object per allocation site to represent all its allocated concrete objects.

finiteness of allocation site -> finiteness of abstract object

Anderson Style

andersen风格是最常见的指针分析, Anderson风格的特点是将分析用一系列子集约束表达.

另一种常见的风格叫Steensgaard’s analysis. 相比于Andersen基于子集约束, Steensgaard风格基于相等约束. 在 Steensgaard风格中, 由于基于相等约束, 可以通过并查集高效的实现.

Context Sensitivity

一个分析是xx敏感的, 则说明该分析可以区分不同的xx. 前述的分析是上下文不敏感的, 因为其堆抽象模型没有区分上下文, 不同上下文中的new site无法进行区分. 所以为了让一个分析是上下文敏感的可以有两种方式: 一种是直接将上下文信息添加到堆抽象中, 比如说一个抽象对象就不止一个位置信息, 还有一个上下文的属性(南大slides里用的就是这种方法). 另一种是新增一种关系来表示一个堆对象处于哪种上下文, 这种方法有一个好处就是, 可以将上下文区分为堆所在上下文(heap context)和函数上下文(context)两种(Yannis那本册子用的就是这种方法).
